It works at 100 docs but falls apart at 10K+.\n\n```typescript\n// BAD — loads entire table into memory\nconst allOrders = await ctx.db.query(\"orders\").collect();\n\n// GOOD — limit results\nconst recentOrders = await ctx.db\n  .query(\"orders\")\n  .withIndex(\"by_createdAt\")\n  .order(\"desc\")\n  .take(50);\n\n// GOOD — use pagination for user-facing lists\nconst page = await ctx.db\n  .query(\"orders\")\n  .withIndex(\"by_createdAt\")\n  .order(\"desc\")\n  .paginate(args.paginationOpts);\n```\n\nAlways pair with an index — without one you're scanning the whole table.\n\n### Hot table / cold table pattern (denormalization)\n\nWhen a table has heavy concurrent reads AND writes, you'll hit transaction conflicts. Split into:\n\n- **Hot table**: receives all writes (inserts, updates)\n- **Cold table** (summary): read-optimized, synced in small batches via a cron\n\n```\nwrites → main table (hot)\n               ↓ batch sync (cron)\nreads  → summary table (cold)\n```\n\nDashboards and listing pages should read from the cold/summary table.\n\n```typescript\n// convex/schema.ts\nexport default defineSchema({\n  orders: defineTable({\n    userId: v.id(\"users\"),\n    total: v.number(),\n    status: v.string(),\n  }).index(\"by_userId\", [\"userId\"]),\n\n  // Read-optimized summary table\n  userOrderSummaries: defineTable({\n    userId: v.id(\"users\"),\n    orderCount: v.number(),\n    totalSpent: v.number(),\n    lastOrderAt: v.number(),\n  }).index(\"by_userId\", [\"userId\"]),\n});\n```\n\n```typescript\n// convex/crons.ts — sync hot → cold in batches\nimport { cronJobs } from \"convex/server\";\nimport { internal } from \"./_generated/api\";\n\nconst crons = cronJobs();\ncrons.interval(\n  \"sync order summaries\",\n  { minutes: 5 },\n  internal.sync.syncOrderSummaries,\n  {},\n);\nexport default crons;\n```\n\n### Document size awareness\n\nIndividual Convex documents have a 1 MB limit, but many medium-sized documents add up fast (e.g., 500 docs × 16 KB = 8 MB scanned per query). Keep documents lean:\n\n- Store large text/blobs in Convex file storage, reference them by `Id<\"_storage\">`\n- Break large nested arrays into separate tables with foreign keys\n- Avoid embedding unbounded lists inside a single document\n\n### Pagination without total counts\n\nComputing a total count (e.g., `\"Page 1 of 47\"`) requires scanning the entire table. Instead, use the `isDone` field from `.paginate()` to show a simple \"Load more\" / `hasMore` indicator:\n\n```typescript\n// Return page + hasMore flag — no expensive count query\nconst result = await ctx.db\n  .query(\"messages\")\n  .withIndex(\"by_channel\", (q) => q.eq(\"channelId\", args.channelId))\n  .order(\"desc\")\n  .paginate(args.paginationOpts);\n\n// result.isDone === true means no more pages\n// result.continueCursor is the cursor for the next page\n```\n\n### Job isolation\n\nBackground work (crons, backfills, syncs) should never compete with real-time user traffic:\n\n- Use `internalMutation` / `internalAction` for all background jobs\n- Give each background job its own cursor / bookmark — don't share state between backfill and live-sync operations\n- Process in
